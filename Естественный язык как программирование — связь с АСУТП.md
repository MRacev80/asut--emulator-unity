## Естественный язык как язык программирования — и связь с АСУТП

### Ключевые идеи
- Естественный язык (русский, английский) может быть **языком программирования**, если упаковать его в строгие контракты/спецификации
- CodeSpeak (от создателя Kotlin) — инструмент: пишешь спецификацию в Markdown → LLM генерирует код + тесты
- Объём поддерживаемого кода сокращается в 6-10 раз
- Роль разработчика: не «пишет код», а **архитектор смыслов** — ревьюит спецификации и тесты
- Эволюция: машинный код → ассемблер → C → Python → **естественный язык в рамках контрактов**

### Резюме
> Естественный язык в формате свободной болтовни — плохой инструмент. Но тот же язык в строгих рамках спецификаций (вход → выход → требования → ограничения) — лучший уровень абстракции. LLM = компилятор, который понимает человеческую речь.

### Прямая связь с АСУТП

Михаил точно заметил: **вся АСУТП — это «если датчик сработал → запустить двигатель»**. Вся логика ПЛК — это условия и действия.

```
АСУТП сейчас:
  IF sensor_level > 200 THEN motor_start := TRUE;

АСУТП через спецификацию:
  # Управление бункером
  ## Условие
  Если уровень в бункере превышает 200 кг
  ## Действие  
  Запустить двигатель шнека
  ## Ограничения
  - Не запускать если аварийная кнопка нажата
  - Таймаут: 30 секунд максимум
```

### Что это меняет для АСУТП

| Сейчас | С LLM-спецификациями |
|--------|---------------------|
| Инженер пишет Structured Text вручную | Инженер описывает логику на русском → LLM генерирует ST |
| Ошибки в коде = авария на производстве | Спецификация на русском → ревью понятно даже технологу |
| Документация отдельно от кода (и устаревает) | Спецификация = документация = код (одно и то же) |
| Главный инженер не читает ST | Главный инженер **может** прочитать спецификацию |
| Изменения в логике = «а кто это писал?» | Изменение в спецификации = Git diff на русском языке |

### Главный инсайт

> **Главное в АСУТП — не код, а архитектура и взаимодействие. Код — это реализация. Архитектура — это смысл.**

Связка:
```
Architecture as Code (модель системы)
    +
Спецификации на естественном языке (логика)
    +
LLM-компилятор (генерация ST/Python/C#)
    =
Инженер проектирует СМЫСЛ, а не пишет код
```

Это совпадает с твоей идеей проекта из статьи про AaC: описать архитектуру Увелки → добавить спецификации логики → LLM генерирует код → ты ревьюишь.
