# Эмулятор АСУТП Unity

## 1. Описание проекта

### Проблема
При разработке АСУ ТП-проектов инженеры пишут ST-код в CODESYS «вслепую»: нет возможности наглядно проверить логику до подключения к реальному оборудованию. Это приводит к багам на объекте, долгому FAT и трудностям при объяснении работы системы заказчику.

### Решение
Инструментальная цепочка, которая позволяет:
- писать ST-код с помощью ИИ по описанию на русском языке
- запускать логику в CODESYS (без железа или с реальным ПЛК)
- видеть работу системы в реальном времени в Unity — как 2D П&ИД схему с живыми значениями
- проводить ручной FAT прямо в Unity
- автоматически генерировать техническую документацию
- реализовывать архитектуру и структуру проекта основываясь на Тех.решении

### Целевая аудитория
- **Инженер-разработчик** — пишет спецификацию, проверяет логику, проводит FAT
- **Заказчик** — видит, как работает его система, ещё до монтажа
- **Новый инженер в команде** — понимает логику через визуализацию, а не через код

### Ключевой результат
Заказчик без технических знаний может наблюдать работу своей системы (открытие клапанов, старт моторов, сигнализации) в Unity — без реального железа в комнате.

## 2a. Детализация Фазы 1 — Проектирование (видение)

> **Статус:** видение, требует проработки реализации

Перед тем как Claude пишет код, инженер должен структурировать знание о системе. Это предотвращает ситуацию «написали — потом переписали» и даёт ИИ достаточно контекста для качественного кода.

### Шаг 1 — Архитектура системы (Architecture as Code)

Используем **LikeC4** (уже исследован в проекте) для описания структуры на уровне C4-модели:

```
Уровень 1 (System):   Установка дозирования Увелка
Уровень 2 (Container): CODESYS PLC / Unity HMI / 1С ERP
Уровень 3 (Component): FB_Shnek / FB_Nasos / FB_Avaria / FB_Dozator
Уровень 4 (Code):      входы, выходы, внутренние переменные каждого FB
```

**Результат:** C4-диаграмма показывает что с чем общается, какие данные передаются. Это основа для таблицы сигналов.

### Шаг 2 — Таблица сигналов (signals.md)

| Тег | Описание | Тип | Направление | Modbus-адрес | Единица |
|---|---|---|---|---|---|
| M101_RUN | Мотор шнека — работает | BOOL | OUT | HR 0 | — |
| M101_FAULT | Мотор шнека — авария | BOOL | OUT | HR 1 | — |
| LT101 | Уровень бункера | INT | OUT | HR 2 | % |
| CMD_START | Команда Старт (из HMI) | BOOL | IN | HR 100 | — |

**Результат:** `signals.md` — единый источник правды для ST-кода, Modbus-конфига и Unity HMI.

### Шаг 3 — Спецификация логики (spec.md)

Описывает поведение каждого функционального блока:

```markdown
## FB_Shnek — Шнек дозирования

### Входы
- bStart: BOOL — команда запуска
- bStop: BOOL — команда останова
- nLevel: INT — уровень бункера (%)

### Логика
1. При bStart=TRUE и nLevel > 5% → запустить мотор (bMotorRun=TRUE)
2. При nLevel < 3% → остановить мотор (защита от сухого хода)
3. При bFault=TRUE → немедленный останов + сигнализация

### Диаграмма состояний
STOP → (bStart & nLevel>5%) → RUN → (nLevel<3% | bFault) → STOP
```

### Шаг 4 — Диаграммы (Mermaid в spec.md)

```mermaid
classDiagram
    FB_Dozator --> FB_Shnek : управляет
    FB_Dozator --> FB_Nasos : управляет
    FB_Avaria --> FB_Shnek : блокирует
    FB_Avaria --> FB_Nasos : блокирует

sequenceDiagram
    HMI->>FB_Dozator: CMD_START
    FB_Dozator->>FB_Shnek: bEnable=TRUE
    FB_Shnek->>FB_Nasos: bNasosReady (после задержки)
```

### Шаг 5 — Ревью спецификации с Claude

До написания кода Claude проверяет spec.md:
- Нет ли противоречий в логике?
- Все ли защиты описаны?
- Правильно ли разбиты функциональные блоки?
- Соответствует ли структура ПОУ архитектуре C4?

- [ ] FR-03: Claude запускает компиляцию проекта и возвращает ошибки в чат

- [ ] FR-06: CODESYS экспортирует переменные через Modbus TCP Slave (встроенная возможность)

- [ ] FR-08: Unity читает Holding Registers с заданным интервалом (по умолчанию 200 мс)

- [ ] FR-10: При потере соединения Unity показывает предупреждение, переподключается автоматически

### 3.3 Визуализация в Unity (2D HMI)

- [ ] FR-12: Числовые значения (температура, давление, уровень) отображаются на схеме в реальном времени
- [ ] FR-13: Индикаторы состояния (работает/стоп/авария) меняют цвет: зелёный/серый/красный
- [ ] FR-14: Кнопки старт/стоп/сброс записывают значение в Modbus-регистр
- [ ] FR-15: Анимированные элементы (вращение шнека, мигание аварии) привязаны к булевым переменным

### 3.4 Документация
- [ ] FR-16: Claude генерирует описание функционального блока по spec.md + ST-коду
- [ ] FR-17: Документ содержит: назначение, входы/выходы (таблица), описание логики, условия аварий
- [ ] FR-18: Формат выходного документа: Markdown (основа) → экспорт в Word или PDF

## 4. Нефункциональные требования

| # | Требование | Ограничение |
|---|---|---|
| NFR-01 | Платформа | Windows 10/11 (CODESYS работает только на Windows) |
| NFR-02 | CODESYS версия | V3.5 SP21 и выше (требование codesys-mcp-toolkit) |

## 5. Архитектура

```
┌─────────────────────────────────────────────────────┐
│  DESIGN-TIME (написание кода)                       │
│                                                     │
│  spec.md  ──►  Claude Code  ──►  codesys-mcp        │
│  (рус.)        (ревью,           toolkit            │
│                генерация)        │                  │
│                                  ▼                  │
│                            CODESYS IDE              │
│                            (create_pou,             │
│                             set_pou_code,           │
│                             compile_project)        │
└──────────────────────────────┬──────────────────────┘
                               │
                    CODESYS запускает ST
                               │  Modbus TCP
                               ▼
┌─────────────────────────────────────────────────────┐
│  RUNTIME (выполнение и визуализация)                │
│                                                     │
│  CODESYS SoftPLC  ──► Modbus TCP ──►  Unity HMI    │
│  (разработка)                         (NModbus C#) │
│                                        │            │
│  Реальный ПЛК    ──► Modbus TCP ──────┘            │
│  (производство)       тот же IP-адрес              │
│                       меняется в настройках         │
└─────────────────────────────────────────────────────┘
```

**Ключевые компоненты:**
- `PLCBridge.cs` — Modbus-клиент в Unity, опрашивает регистры, публикует события
- `VariableConfig.asset` — ScriptableObject: список переменных (имя, адрес, тип, единица)
- `HMIController.cs` — связывает переменные с UI-элементами (лейблы, индикаторы, кнопки)
- `spec.md` — спецификация логики (вход для Claude)
- Проект CODESYS — содержит ST-код, настройку Modbus Slave
