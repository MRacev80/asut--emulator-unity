## Architecture As Code: от рисунка на салфетке к инструменту анализа

### Ключевые идеи
- **Architecture as Code (AaC)** — описание архитектуры кодом (DSL), хранение в Git, автогенерация артефактов
- Отличие от Diagram as Code: DaC = картинки, AaC = **модель** (единый источник правды)
- Инструмент: **LikeC4** (open-source, код → модель → диаграммы → запросы → валидация)
- Компоненты + взаимодействия + метаданные (протоколы, RPS, потоки данных)
- Автоматическая генерация: диаграммы, сводные таблицы API, модели угроз
- Валидация: скрипты проверяют полноту описания (нет протокола? → ошибка в CI/CD)
- Моделирование угроз: границы доверия, потоки данных, механизмы защиты — всё из модели

### Резюме
> Подход Architecture as Code позволяет описать архитектуру системы кодом, положить в Git, и автоматически генерировать диаграммы, таблицы API, модели угроз. Единый источник правды — модель, а не набор разрозненных картинок. Идеально для сложных систем, где нужна согласованность.

### Связь с АСУТП — идея проекта

#### Архитектура SCADA/ПЛК как код
```
model {
    system scadaSystem "SCADA Увелка" {
        container plc_mixing "ПЛК Смешивание (Siemens)" {
            -> scada_termist "Отправляет данные" "Profinet"
            -> bunkers "Управляет дозированием"
        }
        container scada_termist "SCADA Термист" {
            -> erp_1c "Загружает рецептуры" "HTTP/REST" {
                metadata {
                    protocol "Wonderware export template"
                    dataType "Рецептуры, уставки"
                }
            }
        }
    }
}
```

#### Что это даёт для АСУТП:
| Задача | Как решает AaC |
|--------|---------------|
| **Документация проекта** | Единая модель → автогенерация схем взаимодействия, списков сигналов |
| **Тестирование логики** | Описал связи → валидация: все ли сигналы подключены? Все ли протоколы указаны? |
| **Обследование площадки** | Модель «как есть» (as-is) → модель «как будет» (to-be) → diff |
| **Импортозамещение** | Модель Siemens → модель отечественного ПЛК → автоматическое сравнение |
| **Модель угроз (КИИ)** | Границы доверия, потоки данных — обязательно для объектов КИИ |
| **Change Request** | Изменили модель → видно что именно поменялось (Git diff) |
| **Сдача проекта** | Модель = исполнительная документация, всегда актуальная |

#### Связь с линейной алгеброй и нейросетями
- Компоненты системы = **вектора** (набор параметров: протокол, RPS, тип данных)
- Связи = **матрица смежности** (какой компонент с каким взаимодействует)
- Можно применить: кластеризацию (группы связанных компонентов), анализ критических путей
- **ИИ-аудит SCADA**: загрузил модель → ИИ проверяет на типовые ошибки, предлагает оптимизации

### Инструменты
- **LikeC4** — https://likec4.dev/ (open-source, TypeScript, VS Code extension)
- **Structurizr** — https://structurizr.com/ (C4 model, DSL)
- Экспорт: SVG, PNG, PlantUML, Confluence, CSV

### Идея проекта: Моделирование и тестирование логики АСУТП

```
Этап 1: Описать архитектуру Увелки в LikeC4
    - ПЛК, SCADA, 1С, датчики, исполнительные механизмы
    - Протоколы, потоки данных

Этап 2: Добавить валидацию
    - Все сигналы подключены?
    - Все протоколы указаны?
    - Нет дублирования адресов?

Этап 3: Тестирование логики
    - Описать рецептуры как данные в модели
    - Проверять: если рецепт X → какие компоненты задействованы?
    - Сравнение: модель vs реальность (по данным с ПЛК)

Этап 4: Продукт
    - «Архитектурный аудит SCADA» как услуга для заводов
    - Входные данные: схемы + обследование → модель → отчёт с рекомендациями
```
